## Template Guidelines

### Writing Tips

1. **Be Specific**: Always cite file paths with line numbers
2. **Be Evidence-Based**: Include code snippets as proof
3. **Be Balanced**: Show both what exists and what's missing
4. **Be Actionable**: Provide concrete next steps
5. **Be Persona-Aware**: Keep requirements separated by persona

### Coverage Calculation

```
Coverage % = (Fully Met Criteria / Total Criteria) × 100

Example:
- Total criteria: 6
- Fully met: 3
- Partially met: 1
- Not implemented: 2
→ Coverage: 50% (3/6)
```

### Status Icons

Use these consistently:
- ✅ Fully implemented/met
- ⚠️ Partially implemented/met
- ❌ Not implemented/met

### Approach Decision Matrix

| Coverage | Code Quality | Architecture | Approach |
|----------|--------------|--------------|----------|
| 60-100% | Good | Supports extension | **Extend Existing** |
| 30-60% | Issues exist | Some alignment | **Refactor & Enhance** |
| 0-30% | N/A or Poor | Misalignment | **New Implementation** |

### Priority Mapping

| Grade | Priority |
|-------|----------|
| P1 (Critical) | High |
| P2 (Major) | Medium |
| P3 (Minor) | Low |

### Effort Estimation

Consider:
- Lines of code to write
- Number of files to modify
- Complexity of changes
- Testing effort
- Documentation needs

Typical ranges:
- Simple: 1-3 days
- Medium: 3-7 days
- Complex: 7-14 days
- Very complex: 14+ days

Always provide ranges (e.g., "3-5 days") to account for uncertainty.
